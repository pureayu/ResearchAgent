# 项目：文献检索与带引用总结助手

## 0. 项目目标

构建一个面向本地论文 / PDF 文献的轻量系统，支持：

1. 导入和管理本地文献
2. 对文献内容进行检索
3. 基于文献内容问答
4. 回答时附带引用来源
5. 对某个主题生成总结

先不追求复杂平台能力，先把管理、检索、问答、引用、总结这条链路跑通。

## 1. 项目范围

### 第一阶段目标（MVP）

- 支持导入本地 PDF / md / txt 文档
- 支持基础问答
- 支持带引用回答
- 支持主题总结

### 第二阶段目标

- 文献元数据管理
- 文献列表查看
- 单篇文献摘要卡片
- 检索效果对比

### 第三阶段目标

- Web UI
- README 包装
- 简单评测与展示

## 2. 技术栈建议

### 后端

- Python 3.11+
- LightRAG
- FastAPI（后续可选）
- Pydantic
- Uvicorn（后续可选）

### 文档处理

- PyMuPDF / pdfplumber
- 原生 txt / md 读取

### 前端

- 第一阶段：命令行
- 第二阶段：Streamlit（优先）

### 数据存储

- 本地文件
- JSON / CSV / SQLite 任选其一作为 metadata 存储

### 模型接口

- OpenAI 兼容 API
- 或其他可兼容的模型服务

## 4. Phase A：环境与项目初始化

### A1. 初始化项目

- 新建项目目录
- 创建 Python 虚拟环境
- 新建 requirements.txt
- 安装基础依赖
- 安装 LightRAG
- 配置 .env
- 新建基础目录结构

验收标准：

- 项目目录结构完整
- 虚拟环境可正常启用
- 依赖安装成功
- .env 能读取 API Key

### A2. 跑通官方最小 demo

- 阅读 LightRAG README 中最小 demo 部分
- 配置模型 API
- 跑通一次最小文本导入
- 跑通一次最小查询

验收标准：

- 能成功插入测试文本
- 能成功返回一次查询结果
- 确认环境和模型接口是通的

## 5. Phase B：测试文献准备

### B1. 准备测试资料

先不要准备太多，控制在 5 到 10 份：

- 收集 5 到 10 篇 PDF 论文
- 可补充少量 md / txt 笔记文件
- 给文档统一命名
- 建一个简单文档清单

建议字段：

- 文件名
- 标题
- 作者
- 年份
- 标签
- 本地路径

验收标准：

- 有一批可重复使用的测试文档
- 文档信息清晰可追踪

## 6. Phase C：文档导入模块

### C1. 实现文档扫描

- 扫描 `data/raw/` 下的文档
- 识别 PDF / txt / md
- 跳过不支持格式
- 记录扫描结果

验收标准：

- 能枚举出所有待导入文档
- 不支持格式不会导致程序崩溃

### C2. 实现文本抽取

- 读取 txt 内容
- 读取 md 内容
- 读取 PDF 内容
- 对 PDF 文本抽取失败做错误记录
- 统一输出纯文本

验收标准：

- 至少能正确读取 70% 以上测试文档
- 失败文档有日志可查
- 输出文本可用于后续插入

### C3. 写入 LightRAG

- 将抽取文本写入 LightRAG
- 尝试保留文档路径 / 文档名信息
- 导入成功后生成简单记录
- 支持重复导入检测（简单版即可）

文件建议：

- `scripts/ingest_documents.py`

验收标准：

- 多篇文档可成功导入
- 导入后可以查询
- 至少能保留“回答来自哪篇文档”的信息

## 7. Phase D：基础检索问答

### D1. 实现命令行问答

- 从命令行输入问题
- 调用 LightRAG 查询
- 输出回答
- 输出引用来源

文件建议：

- `scripts/query_documents.py`

示例问题：

- 某篇论文的核心贡献是什么？
- 两篇论文的方法有什么区别？
- 某个概念在这些文献里是怎么定义的？

验收标准：

- 能稳定返回回答
- 回答带来源
- 至少能定位到文档名

### D2. 优化引用展示

- 将回答与引用分开显示
- 显示引用文档名
- 尽量显示片段摘要
- 若可行，显示页码或段落信息

验收标准：

- 用户能看出答案依据
- 回答不再像纯黑箱输出

## 8. Phase E：主题总结功能

### E1. 实现主题总结

- 输入一个主题
- 检索相关文献片段
- 基于检索结果生成总结
- 输出总结 + 引用来源

文件建议：

- `scripts/topic_summary.py`

示例主题：

- Transformer 的核心机制
- RAG 的主要挑战
- 多智能体系统的常见设计模式

验收标准：

- 能生成一段结构化总结
- 总结后附带引用
- 总结内容与原文相关，不明显胡编

## 9. Phase F：文献管理能力

### F1. 文献 metadata 管理

- 设计 metadata 存储格式（JSON / CSV / SQLite 三选一）
- 为每篇文献保存基础信息
- 支持标签分类
- 支持年份、标题、标签筛选

推荐字段：

- id
- title
- authors
- year
- tags
- filepath
- abstract（可选）
- notes（可选）

验收标准：

- 知道知识库里有哪些文献
- 能按标签或关键词筛选文献

### F2. 文献列表查看工具

- 列出已导入文献
- 支持关键词搜索标题
- 支持按标签查看
- 显示文档基本信息

文件建议：

- `scripts/list_documents.py`

验收标准：

- 不需要每次问答才能知道库里有什么
- 文献可被浏览和过滤

## 10. Phase G：检索效果优化与评测

### G1. 建立简单评测问题集

准备 15 到 30 个问题，覆盖：

- 单篇论文理解题
- 多篇论文对比题
- 概念解释题
- 主题总结题

### G2. 做基础对比实验

对比内容可包括：

- baseline：默认检索
- 优化版：加 rerank 或调整 query mode
- 比较回答质量
- 比较引用覆盖率
- 比较耗时

记录指标建议：

- 回答可用率
- 引用覆盖率
- 人工评分
- 平均响应耗时

### G3. PDF 解析增强（可选）

- 优化 PDF 文本抽取
- 清理页眉页脚
- 去除参考文献区噪声
- 尝试保留 section 结构
- 提高引用片段自然度

## 11. Phase H：Web UI 与展示

### H1. 用 Streamlit 做最小界面

页面功能：

- 上传 / 选择文献
- 查看文献列表
- 输入问题进行问答
- 查看回答和引用
- 输入主题生成总结

### H2. README 包装

README 需要包含：

- 项目简介
- 核心功能
- 系统架构图
- 安装说明
- 使用方法
- 示例截图
- 后续优化方向
